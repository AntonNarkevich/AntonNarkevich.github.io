Part 3. Acynchrony

What does it mean?

Synchronous code:

  1. Sends request to DB
  2. Waits for response i.e. thread.sleep()
  3. Executes the code after DB call. (processes data)

var db = new DB();
var data = db.getData(); //We sleep here say for 2 s.
console.log(data);

Asynchronous code:

  1. Sends request to DB
  2. Go on with other code
  3. Response from DB come in any time on a callback

var db = new DB();
db.getData(function(err, data) {
    console.log(data);
});
otherWork.do();//Immediately do this work.
               //No need to wait for DB to respond.

This works asynchronously


var fs = require('fs');

fs.readFile('myFile.txt', function(err, data) {
    if (err) {
        console.error(err);
        throw err;
    }

    console.log("Async operation's been completed");
    console.log(data);
}

console.log("Do some other work. Process other request etc.");

More asynchrony


var request = require('request');

for (var i = 0; i < 100; i += 1) {
    request.get("google.com", function(err, data) {
        console.log("Async operaion has been completed");
        console.log("We should process results");
    });

    console.log("Async operation in progress.");
    console.log("We do other work.");
}

How it's implemented?

Выполняет код, который ми пишем только один поток. В отличие от Asp.Net MVC, где под каждый запрос создается поток. Этот потом выполняет какую-то работу. Затем приходит к месту, где начинается операция с OS. Останавливаестся и переходит к выолнению какой-то другой работы из очереди выполнения. (Как в клиентском JavaScript) Как только операция с OS завершена, фаерится событие и в очередь выполнения поступает новая функция-колбек, в которую передается результат выполнения операции.
Т.е. Node.js + OS сами разбираются как эффективным образом испльзовать потоки, чтобы выполнить эти тяжелые задачи максимально эффективно. Основной же процесс завершения их не ждет а за это время обрабатывает как можно большее число запросов.

Gist

We don't wait while file is being read.

We process requests.

We don't waste resources on creating processes.

В этом и состоит основная фишка асинхронности. Это сейчас очень модно. Tornado, async/await в C# и т д.

Part 4. Callbacks

Notice: Node style callback are called errbacks.

Callbacks Rules !important;

  • First argument is error
  • Second argument is the result
  • Never pass both
  • Error should be insanceof Error (troothy)

request.get('http://www.google.com', function (err, response) {
    if (err) {
        return console.error(err);
    }

    console.log('No errors, status is: ' + response.status);
});

Callbacks (Errbacks) !important;

  • Must not be executed on the same tick of the event loop
  • Must be passed as a last argument
  • Returned value is ignored
  • Must not throw any errors/ pass resulting errors
  • Must never be called more than once
https://www.youtube.com/watch?v=UsBng6yfiEQ http://williamdurand.fr/2013/06/03/object-calisthenics/ Он должен быть выполнен на следующем тике. Обязательно. Этого можно достичь с помощью process.NextTick либо setInterval (??в чем разница). Так как, если это, например стрим. То работает он по такому принципу. Создали стрим, подписались на события, отреагировали на события, обработали результат. Если колбек не будет выполнен на следующем тике, то результат вернется до того, как будут повешены обработчики событий. Этому правилу желательно следовать всегда.

global

True global object. If you set:


global.testDirectoryPath = path.join(__dirname, "test");

It will be available anywhere. (From any other modules.)

'window' and 'global' are different


var x = 3; //Doesn't go to global;
y = 4; //Goes to global;
global.z = 5; //Goes to global, evident;

Node's modules are wrapped in a closure, which is evaluated in the this-context of the exports object.

Trend

A lot of modern libs work both: on front and back end.


if (global) {
    //We are in node environment.
}

What is 'this'?

Node's modules are wrapped in a closure, which is evaluated in the this-context of the exports object.

So 'this' is exports.


var assert = require('assert');
assert.equal(this, module.exports);
So var x = 3; Won't create a property on global object. Node's modules are wrapped in a closure, which is evaluated in the this-context of the exports object. So, if you do "var a = 3", then it won't be added to this, global, or exports. But, if you do "this.a = 3" then it'll be added to "this" and "exports". //so var x = 3; //goes to that context as a local variable this.y = 3; //goes to exports z = 3; //goest to global object

exports & module.exports


//Think about exports as about a link:
var exports = module.exports;

Because of this:


exports.myFunction1 = function () {}; //OK
exports = function () {}; //Doesn't change exported object.
                          //Changes only link (similar to pointer)
                          //Be careful.

module.exports = function () {}; //OK
module.exports = { myFunction: function () {} } //OK

So use module.exports.
Because sometimes you'll need:


var m = require('something')();
So if you do exports = { f: function() {} } it won't have an effect; Because still module.exports will be returned. You should use exports.f = function() {}; And at the same time module.exports = { f: function() {} } will have an effect. Also you can: module.exports = function(){}; ... var m = require(module)(); And you cannot: exports = function() {}; ... var m = require(module)(); //Exception.
